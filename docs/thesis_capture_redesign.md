## 1. なぜやるか（実ユーザー検証由来）

改善ループの「実ユーザー検証」軸で、実ユーザー（naotaro）が **Thesis→Verdict の検証（答え合わせ）をほとんど回していない**ことが判明。
本人の自己診断は3点：

1. 検証の期日がなかなか来ない
2. 仮説が一文でまとまらない
3. 銘柄分析が浅い＝「何がどうなればどう動くか（④数値への接続）」を言語化できていない

診断で分かった核心：**「一文でまとまらない」は文章力の問題ではなく、①テーマ→②セクター→③銘柄固有性→④数値→⑤株価→⑥反証→⑦観測点 という因果の鎖を歩いていないから、結論の一文が結晶化しようがない**。詰まる段は本人の実感で **④数値への接続**。

さらに設計上の合意：
- ④を「利益をモデル化する」と捉えると重く、負担増→継続利用者が減る。証券会社/アナリストの仕事であり `product_context` の判断軸「❌ 証券会社が既にうまくやっていること」に当たる。
- **記録ツールにとっての④は、数字を計算することではなく「答え合わせで何を見るか」を1つ先にコミットすること**。magnitude は予測せず、実際の結果に教えてもらう（＝成長OSの「予想→結果→検証→学び」そのもの、spaced repetition の投資版）。
- ただし「数字＋向き」に固定すると **イベント型・長期定性テーマ型の仮説が歪み、幅を殺す**（使い勝手の毀損）。→ 一般化と足場化で回避（§3）。

**AI分析は使わない前提**（本人の明示条件）。目印はユーザー自身が選ぶ。

## 2. 現状（`_thesis_form.html` / `ThesisForm` / `Thesis`）

入力順：`claim`(必須3行) → `basis_tags` → `basis`(4行) → `horizon`＋`review_due_date`(既定6ヶ月) → `worst_case`(任意・最下部)。

現状が診断を体現している3問題：
- **主役が「完成した主張文」**：鎖を歩けていない人に結晶を要求＝詰まる場所そのもの。
- **「確認の目印」欄が無い**：④⑦を具体化する一番安いヒンジがどこにもなく、free text に埋もれる。
- **`worst_case` が最下部・任意**：一番大事な反証条件がスキップされる位置。

---

## 3. 確定設計

**思想：キャプチャを「小論文」から「賭け（答え合わせで何を見るか先に決める）」へ。**

| 要素 | 設計 |
|---|---|
| **確認の目印（checkpoint）** | 答え合わせで見るものを1つコミット。**数字に限らない**：財務数字・株価/PER・KPI（販売台数/シェア）・イベント結果・定性チェックも可。これが入力の主役。 |
| 向き（checkpoint_direction） | 目印に応じ「上/下/横ばい」or「そうなった/ならなかった」。**必須にしない**（イベント型/定性型のため） |
| claim | 目印＋向き＋期日から**自動生成**を既定にする（例「資金利益が次決算までに上がる」）＝一文問題を消す。手直しは可。 |
| worst_case（反証条件） | **穴埋めで前へ**移動（スキップさせない） |
| horizon | 既定を「**次の決算まで**」に。長期含む既存選択肢は残す（**ロックしない**） |
| 逃げ道 | ガイド付き「賭け」ビルダーは**足場**。「**自由に書く**」モードを常設し、定性・長期テーマ型の幅を殺さない |
| 引き算 | `basis`/根拠タグは「**もっと書く**」に畳み、主フォームを短縮（負担減） |

**効く理由**：④を「モデル化」→「目印を1つ選ぶ」に軽量化、①を次決算既定で解消、②を自動生成で解消。負担は増やさず（むしろ主フォーム短縮）、仮説の幅は逃げ道と目印一般化で保つ。

### 幅の検証（型 × 適合）
業績/バリュエーション/マクロ経由/リスク型＝目印＋向きで✅。イベント型＝目印（結果）＋「そうなった/ならなかった」で対応。長期・定性テーマ型＝自由記述＋長期horizon＋定性チェックポイントで対応（数字に落とさない）。

## 4. スキーマ変更（決定：構造化する＝分岐1を採用）

**理由**：検証（Verdict）時に「あなたが見張ると決めた目印＝◯◯。実際どうだった？」と**機械的に問い返せる**ことが、「答え合わせが回らない」の核を直接解く。テキスト埋もれ（分岐2）ではこの問い返しができない。

`Thesis` に追加（案・実装時に確定）：
- `checkpoint`：CharField（確認の目印。答え合わせで見るもの）。任意（自由記述モードでは空可）。
- `checkpoint_direction`：CharField choices（up/down/flat/happened/not_happened 等）。任意。

既存 `worst_case`・`horizon`・`review_due_date` はそのまま活用。`horizon` の既定値を `next_earnings` に変更（現状 `6m`）。

---

## 5. 検証時の問い返し（Verdict 連携）

`thesis_verify` で、保存済みの `checkpoint`＋`direction` を提示し「実際は？」を一次質問にする。
これにより答え合わせが「白紙から書く」から「コミット済みの目印を答える」に軽くなる。
`metrics.py`（的中/勝ちの意味定義の正本）とは疎結合を保つ（checkpoint は観測点であり、勝敗判定ロジックは metrics 側の責務）。

## 7. 明示的な非目標

- **AI分析は使わない**（目印はユーザーが選ぶ）。
- magnitude（利益のモデル化）を要求しない。
- 既存の自由記述の幅を奪わない（足場であって強制ではない）。
- `AggregateService`（FIFO損益）には触れない（意思決定の質と損益は疎結合）。
